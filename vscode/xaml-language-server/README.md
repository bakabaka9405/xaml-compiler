# XAML Language Server

XAML language support for WinUI / Windows App SDK projects, powered by the Microsoft XAML Compiler.

## Features

- Syntax highlighting for `.xaml` files
- Diagnostics (errors and warnings) via the XAML Compiler
- Hover information
- Code completion

## Prerequisites

- [.NET SDK](https://dotnet.microsoft.com/download) (for the language server backend)
- [Node.js](https://nodejs.org/) 18+
- [Bun](https://bun.sh/) (package manager)

## Development

```bash
# Install dependencies
bun install

# Build the extension
bun run compile

# Watch mode (auto-rebuild on changes)
bun run watch

# Run tests (requires .NET server to be built first)
bun run test

# Lint
bun run lint

# Package for distribution
bun run package
```

Press **F5** in VS Code to launch the extension in the Extension Development Host.

## Build Pipeline

The extension consists of two parts:

1. **TypeScript client** (`src/extension.ts`) — bundled with esbuild into `dist/extension.js`
2. **.NET LSP server** (`server/`) — built via `scripts/prepare-server.js`

The `vscode:prepublish` hook runs the full pipeline: .NET server build → type checking → linting → esbuild bundling.

## Project WinMD Discovery

The extension can discover project-local WinMD files produced by [WinUI3 xmake](https://xmake.io) builds, enabling
accurate `x:DataType` validation for local project types.

### How It Works

1. **Path pattern**: The extension scans `build/.gens/<target>/winmd_merged/<namespace>.winmd` under the workspace root.
   - `<target>` is the xmake target name (e.g., `demo.hello`)
   - `<namespace>` is the project namespace declared via `set_values("winui3.namespace", "...")` in xmake.lua

2. **Scan triggers** (no file watcher):
   - Extension activation (startup)
   - Opening a `.xaml` file
   - Saving a `.xaml` file
   - Typing changes do **not** trigger rescans

3. **Matching strategy**:
   - Exact namespace match is preferred (e.g., namespace `hello` → `hello.winmd`)
   - When no exact match exists, all discovered WinMD files are loaded as ambiguous candidates
   - Duplicate paths are case-insensitively deduplicated

### Scope and Limitations

- **Diagnostics only**: Project WinMDs are used for `x:DataType` validation and type-not-found diagnostics.
  They do **not** extend completion, hover, or go-to-definition capabilities.
- **Graceful degradation**: If `build/.gens` does not exist (e.g., before first build), the extension
  operates normally using global reference assemblies only — no errors are shown.
- **No automatic builds**: The extension does not trigger xmake builds or any other build commands.
- **WinUI3 xmake only**: The discovery path pattern is specific to winui3-xmake project layouts.
  MSBuild, CMake, or other build system outputs are not supported.
- **No file watcher**: The extension does not use filesystem watchers to detect new build outputs.
- **No persistent cache**: WinMD references are not persisted across LSP restarts.

### Requirements

- A [WinUI3 xmake](https://github.com/zhang-ji-chao/winui3-xmake) project structure
- `set_values("winui3.namespace", "xxx")` declared in the project's xmake.lua
- Build output in `build/.gens/<target>/winmd_merged/` (generated by `xmake build`)
